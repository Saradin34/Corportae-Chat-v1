# Corporate Chat v2.0 PRO 💬

Полнофункциональный корпоративный мессенджер реального времени.
Telegram-подобный дизайн, тёмная/светлая тема, адаптив, WebSocket,
PostgreSQL, Redis, Nginx — всё в Docker. Запуск одной командой.

```bash
python setup.py
```

## 🔐 SSO / Active Directory (LDAP)

Поддерживается вход через **on-prem Active Directory** по LDAP/LDAPS с
прямым bind к контроллеру домена, **JIT-провижинингом** (пользователь
создаётся при первом успешном входе) и **маппингом ролей по AD-группам**.

### Быстрая настройка «только AD»

В `.env`:

```env
ALLOW_LOCAL_AUTH=false
LDAP_ENABLED=true
LDAP_SERVERS=ldaps://dc1.company.local:636,ldaps://dc2.company.local:636
LDAP_USE_SSL=true
LDAP_BASE_DN=DC=company,DC=local
LDAP_DOMAIN=company.local
LDAP_NETBIOS=COMPANY
# члены этой AD-группы становятся администраторами приложения
LDAP_ADMIN_GROUP=CN=ChatAdmins,OU=Groups,DC=company,DC=local
# (опц.) только члены этой группы могут входить
LDAP_REQUIRED_GROUP=CN=ChatUsers,OU=Groups,DC=company,DC=local
```

Затем `docker compose up -d --build`. На странице входа появится плашка
«🔐 Вход через Active Directory», локальная регистрация будет скрыта.

### Как это работает
1. Пользователь вводит доменный логин (`ivanov`, `COMPANY\ivanov` или
   `ivanov@company.local`) и пароль.
2. Backend делает **bind** к контроллеру домена — успешный bind = верный пароль.
3. Читаются атрибуты (`displayName`, `mail`, `memberOf`).
4. Если есть `LDAP_REQUIRED_GROUP` — проверяется членство (allow-list).
5. Пользователь создаётся/обновляется; роль = `admin`, если он в `LDAP_ADMIN_GROUP`.
6. **Синхронизация групп**: группы/OU из `memberOf` зеркалируются в групповые
   чаты приложения (`LDAP_SYNC_GROUPS=true`), и пользователь автоматически
   добавляется в соответствующую группу. Так каждый отдел AD становится
   готовым групповым чатом. Фильтры: `LDAP_GROUP_FILTER` (что включать),
   `LDAP_GROUP_EXCLUDE` (что игнорировать, напр. `Domain Users`).
7. Выдаётся обычный JWT — дальше всё работает как раньше.

### Режимы
| Сценарий | `LDAP_ENABLED` | `ALLOW_LOCAL_AUTH` |
|---|---|---|
| Только AD (рекомендуется для корпоратива) | `true` | `false` |
| Гибрид (AD + локальные «аварийные» аккаунты) | `true` | `true` |
| Только локально (по умолчанию) | `false` | `true` |

> В гибридном режиме при неуспехе AD выполняется попытка локального входа —
> удобно для аварийного `admin`. Пароли AD-пользователей **нельзя** менять
> внутри приложения (управляются в домене).

### Безопасность LDAP
- Поддержка **LDAPS** (636) и **StartTLS** (389).
- Экранирование LDAP-фильтров (защита от LDAP-инъекций).
- Несколько контроллеров домена (failover через `ServerPool`).
- Опциональный сервисный аккаунт для чтения каталога.

## 🔒 Безопасность (и почему антивирус/Dr.Web не ругается)

- **Ноль внешних запросов.** Всё self-hosted: эмодзи рендерятся локально
  системным шрифтом (раньше грузился CDN Twemoji — это и вызывало
  предупреждения Dr.Web). Фронтенд, шрифты, иконки — только с вашего сервера.
- **Same-origin.** Фронтенд и API на одном домене через nginx — никаких
  cross-origin запросов. CORS закрыт (без `*`), при необходимости — белый
  список через `CORS_ORIGINS`.
- **Заголовки безопасности** (nginx): `Content-Security-Policy` (только
  `'self'`), `X-Content-Type-Options: nosniff`, `X-Frame-Options: SAMEORIGIN`,
  `Referrer-Policy`, `server_tokens off`, а в HTTPS-режиме — HSTS.
- **HTTPS** для боевого использования (см. `docker-compose.https.yml`) —
  пароли AD и токены не ходят открытым текстом.
- Пароли через bcrypt, JWT, защита LDAP-инъекций, LDAPS для AD.

> Если корпоративный антивирус ранее блокировал запросы — после этих
> изменений приложение работает полностью изнутри вашей сети и не обращается
> в интернет, поэтому поводов для срабатываний нет.

## 🔔 Уведомления (работают на всех ПК)

- Всплывающие уведомления приходят на **любой** компьютер, где открыт чат,
  когда сообщение пришло **не в активный чат** или окно свёрнуто/не в фокусе.
- Доставка в реальном времени по WebSocket каждому участнику чата.
- Для надёжности зарегистрирован локальный **service worker** (`/sw.js`,
  без сети). Уведомления требуют **HTTPS или localhost** — в проде включите
  HTTPS (см. ниже). В десктоп-приложении разрешение выдаётся автоматически.
- Настройки → Уведомления: звук, всплывающие, предпросмотр текста.

## ❗ `InvalidPasswordError` / `password authentication failed`

Это **самая частая** проблема при повторных запусках. PostgreSQL задаёт
пароль пользователя **только один раз** — при первой инициализации тома
данных. Если том `pgdata` остался от предыдущего запуска с другим паролем,
backend не сможет подключиться (в логах — `InvalidPasswordError`).

**Решение — сбросить том БД:**

```bash
docker compose down -v        # -v удаляет том с устаревшим паролем
docker compose up -d --build
docker compose logs -f backend
```

`setup.py` теперь **определяет эту ошибку автоматически** и предлагает
сбросить том в один клик.

## 🔁 Если backend «unhealthy» при первом запуске

На самом первом `up` PostgreSQL инициализирует данные и кратковременно
поднимает временный сервер — `pg_isready` может ответить «готов» на пару
секунд раньше, чем БД реально примет подключение приложения. Раньше из-за
этого backend падал на старте (`dependency failed to start`).

Теперь backend **повторяет** подключение к БД с задержкой (до 30 попыток),
а не падает. Просто запустите снова:

```bash
docker compose up -d            # или повторно python setup.py
docker compose logs -f backend  # видно строки "Database not ready ... retry"
```

Контейнер сам дождётся БД и станет healthy.

## 🐛 Исправленные ошибки из прошлой версии

- **`event.target.closest is not a function`** — эмодзи-пикер переписан:
  обработчики используют `currentTarget`, делегирование защищено проверкой
  `typeof target.closest === "function"`, инициализация выполняется только
  при открытии чата (а не на `DOMContentLoaded`).
- **`502 Bad Gateway` на `/api/auth/login`** — nginx стартует только после
  `healthcheck` бэкенда (`depends_on: condition: service_healthy`), upstream
  указывает на сервис `backend:8000`, добавлен `proxy_next_upstream`.
- **Конфликт Python/JS** — бэкенд всегда работает на Python 3.11 в Docker;
  `setup.py` написан на чистой stdlib и совместим с любой версией Python.
- **`dependency failed to start` / backend unhealthy** — backend больше не
  падает, если БД ещё не готова: добавлен retry-цикл подключения с backoff,
  а `start_period` healthcheck увеличен (см. раздел выше).
